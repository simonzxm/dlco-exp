\section{寄存器堆设计}

实验所用的寄存器堆支持异步读和同步写，具体来说：

\begin{itemize}
    \item \textbf{异步读}：读取直接通过 \texttt{assign} 语句实现，无需时钟沿信号
    \item \textbf{同步写}：写入操作在时钟上升沿触发，使用 \texttt{always @(posedge clk)} 块实现
\end{itemize}

\section{RAM 设计（IP 核）}

RAM 部分利用 Vivado 的 Block Memory Generator IP 核生成一个 16$\times$8 的单口存储器。由于本项目不使用 Vivado GUI，IP 核通过 Tcl 脚本自动创建：

\begin{lstlisting}[language=Tcl]
# Create in-memory project and generate IP
create_project -in_memory -part $PART
create_ip -name blk_mem_gen -vendor xilinx.com -library ip -version 8.4 \
    -module_name blk_mem_gen_0 -dir $IP_DIR
set_property -dict [list \
    CONFIG.Memory_Type {Single_Port_RAM} \
    CONFIG.Write_Width_A {8} \
    CONFIG.Write_Depth_A {16} \
    CONFIG.Read_Width_A {8} \
    CONFIG.Enable_A {Always_Enabled} \
    CONFIG.Write_Width_B {8} \
    CONFIG.Read_Width_B {8} \
    CONFIG.Register_PortA_Output_of_Memory_Primitives {false} \
    CONFIG.Load_Init_File {true} \
    CONFIG.Coe_File [file normalize ./ram_init.coe] \
] [get_ips blk_mem_gen_0]
generate_target all [get_ips blk_mem_gen_0]
synth_ip [get_ips blk_mem_gen_0]
\end{lstlisting}

初始化使用 \texttt{.coe} 文件，将 16 个单元分别初始化为 \texttt{0xF0}--\texttt{0xFF}。

与寄存器堆不同，IP 核生成的 RAM 的读写都是同步的，在时钟上升沿完成数据更新。

\section{顶层模块设计}

顶层模块将寄存器堆和 RAM 共用同一组地址开关和时钟按钮，通过一个选择开关区分写入目标。

开发板按钮 \texttt{BTNC} 直接作为两个存储器的时钟信号，每按一次产生一个时钟周期。由于 \texttt{BTNC} 并非专用时钟引脚，XDC 中需设置：
\begin{lstlisting}[language=Tcl]
set_property CLOCK_DEDICATED_ROUTE FALSE [get_nets BTNC_IBUF]
\end{lstlisting}

\section{实验结果与分析}

我们通过 Vivado 的 Report Utilization 工具分析了综合后的资源利用情况，使用以下命令生成报告：

\begin{lstlisting}[language=Tcl]
report_utilization -file $BUILD_DIR/reports/utilization.txt
report_utilization -hierarchical -file $BUILD_DIR/reports/utilization_hierarchical.txt
\end{lstlisting}

\subsection{读写时钟周期差异}

\begin{itemize}
    \item \textbf{寄存器堆}：写入需要一个使用周期，读取需要零个时钟周期。
    \item \textbf{RAM（IP 核）}：读取和写入都需要一个时钟周期。
\end{itemize}

\subsection{综合后的实现方式}

在 Utilization by Hierarchy 表中，我们发现：

\begin{itemize}
    \item \textbf{寄存器堆} (\texttt{regfile\_inst}) 使用了 8 个 LUTRAMs，实现方式为分布式 RAM。
    \item \textbf{RAM（IP 核）} (\texttt{ram\_inst}) 使用了 1 个 RAMB18，说明它被实现为 Block RAM。
\end{itemize}

\subsection{资源消耗统计}

根据 Utilization by Hierarchy 表，我们可以看出，\textbf{寄存器堆} (\texttt{regfile\_inst}) 使用了 8 个 LUTRAMs 和 5 个 Logic LUTs，没有使用任何专用的 Block RAM 和 FF。

而 \textbf{RAM（IP 核）} (\texttt{ram\_inst}) 使用了 1 个 RAMB18，也就是 0.5 个 Block RAM Tile (根据 Memory 表)，没有使用任何查找表和 FF。

\section{实验总结与反思}

由于我的笔记本用不了 Vivado，平时一般是 SSH 到寝室的 Linux 电脑上使用的，所以使用 Vivado GUI 的体验不要好。
在非 GUI 环境下创建 IP 核是非常麻烦的，这部分我做了很久才搞定，也借助了一些外界的帮助。

不过后面的综合分析部分，我感觉在非 GUI 环境下还是很方便的，生成的报告文件很适合阅读。

对于寄存器堆和 RAM 的实现方式，说实话我之前并没有搞的太懂，现在看了分析报告我才知道其中具体的原理。
之前我只对寄存器和 RAM 有一个初步的认识，听说过寄存器比 RAM 快，但不知道为什么。
感觉还是现在上手自己实现了一个，才真正理解了其中的区别。

做这个实验已经是一个月前的事情了，现在回过头来写报告，感觉自己像是又把整个实验重新做了一遍。
